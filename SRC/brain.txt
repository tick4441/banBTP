How different Windows store their boot splashes This document details my findings for how a few select major versions of Windows store the graphics that accompany their boot screens. This is NOT a tutorial for replacing boot splashes - this is technical information intended mostly for entertainment purposes, or for people interested in implementing their own hobby boot-graphic-replacement programs. Windows 1.01 The graphics are stored in WIN.COM in the last 2412 bytes of the file. There are two 268x18 2 bpp monochrome bitmaps that come immediately one after another. Each is 1206 bytes. The bitmaps have no header, no padding, and no color table, just raw data, scanlines one after another. Scanlines go top-to-bottom and bits represent pixels left-to-right. A 11 bit pair represents a white pixel, a 00 bit pair represents a dark blue pixel. Iâ€™m unsure what a 10 or 01 bit pair does or why the images are 2 bpp in the first place since they are monochromeâ€¦ The two bitmaps combine at runtime to make a single graphic. The scanlines of the two are interleaved (one from the first, then one from the second, then one from the first againâ€¦ etc) In other words, to make the two 18-scanline bitmaps from a single 36-scanline bitmap, take out every even scanline and stack them up, then do the same for the odd scanlines. For further research: In WIN.COM, Iâ€™m unsure if the first bitmap is for the even or odd scanlines of the final image. If you are so inclined, the text that appears on the boot screen is also stored in WIN.COM as plaintext, though Iâ€™m unsure if the strings are counted or null-terminated, and if a possible counter is in the file or not. (how it appears) (how itâ€™s stored in memory) Windows 2.03 Works in exactly the same way as 1.01, though the stored graphic is different. Windows 3.1 The graphics are once again stored in WIN.COM. This time, the whole boot splash is a proper bitmap with a standard 40-byte BITMAPINFOHEADER header, starting at offset 0x43F0 and continuing to the end of the file. The bitmap is 355x425, at 4 bpp with a 16 color table (though only 8 colors seem to be used, the rest being all black), with BI_RLE4 compression. The exact colors in the table are: [indices 0-8] 0x000000 â–ˆ [index 9] 0x008000 â–ˆ [index 10] 0x3f87a3 â–ˆ [index 11] 0x0000ff â–ˆ [index 12] 0xff0000 â–ˆ [index 13] 0x3da0c7 â–ˆ [index 14] 0x77bed9 â–ˆ [index 15] 0xffff00 â–ˆ Since it is a regular bitmap, it can be saved and edited with standard image tools. Plus, the exact bitmap format does not need to be respected - you can change the color table, use all 16 colors, even change the dimensions of the image! You must keep it at 4 bpp, however, otherwise it will display incorrectly. Windows 95 (OSR A) The boot graphic is embedded in IO.SYS, a hidden system file in the root of the C: drive (or wherever the OS was installed). It is obscured by a custom proprietary compression scheme. The compressed data spans from offset 0x1B210 to offset 0x2B17A. With the help of this Windows ME IO.SYS decompressor, I have been able to fully reimplement the decompression and compression algorithms for at least this particular bitmap. However, none of this is necessary to use a different startup graphic. If a bitmap of a specific format is included in the root of the C: drive with the name LOGO.SYS, Windows 95 will use that graphic instead of the one embedded in IO.SYS. LOGO.SYS must be a 320x400 8 bpp bitmap. Even though sources on the internet say the bitmap is RLE compressed, it does not have to have BI_RLE8 compression, and in fact the embedded IO.SYS graphic is not RLE compressed! Note that the bitmap is taller than it is wide. At runtime, it is horizontally stretched from a 4:5 ratio to a 4:3 ratio, (i.e., from 320x400 to 640x480). The animation of the loading stripes scrolling across the bottom of the image is partially defined by the bitmap itself. The last 20 colors in the color table will be rotated through at runtime, so any part of the bitmap using those colors is animated. This animation feature relies on a very rarely used field, biClrImportant (4 bytes at file offset 0x32) being set in the bitmap header. The number in this field is the number of colors that stay fixed, so for instance if it is set to 236, then the first 236 colors appear as normal and the final 256 - 236 = 20 colors will be cycled through. If itâ€™s set to 256, no animation will occur at all. If itâ€™s set to 1, you will get a rainbow rave dance party. Most image editors seem to just discard this information entirely because it is practically never used, so after saving your bitmap with something like GIMP, youâ€™ll have to edit it with a hex editor to restore the biClrImportant field. Windows 98 (First Edition) The graphics format, animation, and LOGO.SYS trick all work the same as in Windows 95. IO.SYS is laid out slightly differently. The compressed data appears to be at offset 0x1E030 - 0x2E285 this time. The stored graphic is also obviously different, though the format is the same. Windows ME Once again, the graphics format, animation, and LOGO.SYS trick all work the same as in Windows 95. However, the IO.SYS situation is somewhat different. Rather than just compressing the bitmap in IO.SYS as in 9x, the entire file is compressed, using a very similar compression scheme as for the 9xsâ€™ logos. A tool exists for uncompressing this file here (note the expired https certificate, but the program is safe at the time of writing this). Once uncompressed, the bitmap is indeed easily discoverable and replaceable. The uncompressed (and edited) IO.SYS can be used in place of the compressed one, Windows ME will handle it all the same. Windows 2000 (Professional RTM) The boot graphic is stored as an embedded resource in NTOSKRNL.EXE. This is a standard Windows PE executable and the graphic is embedded in a completely standard way, so there are many tools available that can easily access and modify the graphic, such as Resource Hacker. In the PE file, the bitmap is identified with an ID of 1. The graphic itself is once again a Windows bitmap. It is 640x480, at 4 bpp with a 16 color table, all 16 colors being used, using BI_RLE4 compression. Windows XP (Professional SP2 x86) The boot graphic can be found in the exact same place as Windows 2000 (embedded PE resource in NTOSKRNL.EXE, Bitmap with an ID of 1) Once again, it is 640x480 at 4 bpp. It appears completely black because the 16 entries of the color table are all 0x000000, however, all the bitmap data is there, and can be seen if the color table is manually replaced.
